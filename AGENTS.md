# AI 助手操作说明

这份笔记用于让 AI 助手快速理解当前 Obsidian vault 的结构和操作约定。

## Vault 概况

- `README.md` 是全局入口。
- 笔记多使用中文标题，按学科和主题分层组织。
- `_resources/` 是全局资源目录，用于集中存放图片、附件等非笔记页资源。

## 读取与新增

1. 优先查看 `README.md`。
2. 进入具体领域前，先读取对应目录下的 `总览`、`概括`、`简介` 或 `README.md`。
3. 修改或新增笔记前先搜索相关主题，避免重复创建相近内容。
4. 新增笔记尽量放入已有分类目录，不直接放在根目录。

## 资源文件约定

- 所有非笔记页资源文件统一放在 vault / repo 根目录的 `_resources/` 下，包括图片、截图、PDF、表格、音视频和其他附件；不要在各主题目录下新建 `_images`、`images`、`assets` 等资源目录。
- 新增或移动资源后，必须同步更新引用它的笔记链接。
- 资源文件名应简短、可辨识，尽量保留中文语境；若不同目录迁入 `_resources/` 时出现同名文件，使用主题或目录名前缀区分，避免覆盖。
- 笔记中引用 `_resources/` 内资源时，使用标准 Markdown 链接或图片语法，并写成根相对、URL 编码路径，例如 `![](/_resources/%E7%A4%BA%E4%BE%8B.png)`。
- `_resources/` 只存放资源文件，不放知识笔记、目录总览或临时说明文档。

## 写作约定

- 使用清晰的 Markdown 标题层级。
- 同一主题尽量保持一篇主笔记，必要时再拆分子主题。
- 保留中文语境和已有命名风格。
- 历史、科学等内容优先写结构化摘要，再补充细节。
- 修改或重构笔记时，不丢失原有事实、例子、图片、链接和辨识要点；可以重新组织、归类、补充上下文。
- 内部笔记和资源链接统一使用标准 Markdown 链接，不使用 Obsidian 双中括号 wiki 链接。
- 内部链接一律写成相对于 vault / repo 根目录的根相对路径：以 `/` 开头，不使用当前目录相对路径或 `../`。
- 内部链接路径必须 URL 编码，保留路径分隔符 `/`，并避免双重编码。示例：`[明](/%E4%BA%BA%E6%96%87%E7%A7%91%E5%AD%A6/%E5%8E%86%E5%8F%B2-%E4%B8%AD%E5%9B%BD/%E6%9C%9D%E4%BB%A3/%E6%98%8E/README.md)`；目录名 `晋&赵魏韩` 应编码为 `%E6%99%8B%26%E8%B5%B5%E9%AD%8F%E9%9F%A9`，不要写成 `%E6%99%8B%2526%E8%B5%B5%E9%AD%8F%E9%9F%A9`。
- 转换旧链接时，原目标 `README.md` 表示“当前笔记所在目录的 README.md”，不能误写成根目录 `/README.md`。

## 体系类目录整理约定

整理文字系统、语言系统、生物分类、自然科学分类、技术谱系等“层级 / 演化 / 派生关系”明显的笔记时，优先采用“目录 README 导览 + 下级目录 README”的嵌套结构。

- 顶层主题目录统一建立 `README.md`，承担入口和总览功能：先给出体系范围、分类原则和关键易混点，再用 Mermaid 图展示主要层级或演化关系，并用表格列出核心子系统。
- 每个重要系统、分支、阶段或类别独立成目录，并在该目录下建立 `README.md`；目录层级应直接表达“上位系统 → 下位系统 / 派生系统 / 子类”的关系。
- 具体节点笔记一般使用以下结构：`# 名称`、可选的 `## 时间` / `## 范围`、`## 概括`、`## 演变关系` / `## 分类关系`、`## 说明`、`## 上级`、可选的 `## 子系统` / `## 子类`、`## 参考资料`。
- 体系图统一使用 Mermaid 代码块，不使用纯文本箭头、截图或不可维护的图片流程图。
- Mermaid 节点标签包含空格、斜杠、括号、标点或其他特殊字符时，使用 quoted label：`NodeId["显示文本 / 别称"]`。
- 普通继承、派生、分化、影响、传播、演化关系按“来源节点 → 后继节点”书写；如果关系有争议或只是可能影响，应在边标签或正文中写明“可能影响”“受影响”“谱系关系有争议”，不要写成确定直系继承。
- 上级 README 只保留总览、主线、关键差异和链接；细节事实、例子、争议、术语辨析、具体用途和地区差异放入对应子目录 README。
- 从旧笔记、raw data、图片、流程图或清单转写内容时，不写“原图”“图片参考”“raw data”“原数据”等来源性话语；直接整理成知识笔记，并确保原有事实、例子、辨识点和链接不丢失。
- raw data 中的 HTML 样式、颜色、高亮、缩进清单应转成标准 Markdown 标题、列表、表格和 Mermaid 图；不要把 `<span>` 等展示性标签带入整理后的笔记。
- 层级关系必须按知识事实修正，不机械照搬旧目录。例如文字系统中，圣书体可作为古埃及文字体系和原始西奈字母的重要图形来源；原始西奈字母下再整理腓尼基字母、南阿拉伯字母等分支；古希伯来字母、亚兰字母、希腊字母更适合放在腓尼基 / 迦南字母分支下。
- 语言系统整理时，目录层级优先表达“语系 → 语族 → 语支 → 语言 / 方言群”的关系；每层 README 应说明分布、代表语言、内部分类和争议分类，避免把文字、民族、国家和语言谱系混为一谈。
- 自然科学体系整理时，目录层级优先表达学科分类、尺度层级、演化关系或分类单元；每层 README 应说明分类标准、代表对象、关键机制和例外情况，避免只堆名词清单。
- 体系类笔记可适当补充外部资料来修正旧笔记错误或补足缺漏；引用外部资料时，优先使用权威百科、专业机构、教材或论文来源，并在 `## 参考资料` 中列出链接。
- 整理完成后，为新增或修改的笔记补齐 Obsidian 属性，至少包含 `status: 已整理` 和 `title: 笔记标题`。

## 王朝更迭类目录整理约定

整理某个文明、地区、国家或半岛的王朝更迭、政权演变类笔记时，优先采用“目录 README 导览 + 阶段独立笔记”的结构。

- 目录总览文件统一命名为 `README.md`，只承担入口和总览功能：保留一张 Mermaid 演进流程图，并用表格按历史存在顺序列出“顺序、名称、时间、简要概括”。
- 王朝更迭、政权演变、文明发展等“演进图 / 发展图”统一使用 Mermaid 代码块，不使用纯文本箭头、截图或非 Markdown 可维护格式。
- 整理 `人文科学/历史-中国/民族` 时，每个民族专属笔记、每个民族大类 `README.md`、每个民族小类 `README.md` 都必须保留一张与该层级内容对应的 Mermaid 民族演进图；大类图覆盖该大类主要分支，小类图覆盖该小类内部节点，民族专属笔记图覆盖该民族自身的起源、分化、迁徙、政权继承或后续融合线索。
- Mermaid 演进图的方向规则：普通发展、继承、分支、迁徙、南渡等关系按“来源节点 → 后继节点”书写；灭亡、吞并、征服类关系按“被灭 / 被吞并 / 被征服者 → 灭亡者 / 吞并者 / 征服者”书写。例如“金灭辽”写作 `辽 -->|1125 金灭辽| 金`，“蒙古灭西夏”写作 `西夏 -->|1227 蒙古灭西夏| 蒙古`。
- 多方参与的灭亡关系，应以主要灭亡者为主线，同时可用另一条边表示重要参与者。例如“蒙宋灭金”可写 `金 -->|1234 蒙古灭金| 蒙古`，并补充 `金 -->|1234 南宋联蒙攻金| 南宋`。
- Mermaid 节点标签包含空格、斜杠、括号、标点或其他特殊字符时，使用 quoted label：`NodeId["显示文本 / 别称"]`。
- Mermaid 虚线关系不要写成 `A -.文字. B`，这是已知会触发解析错误的写法；无标签虚线写 `A -.-> B`，需要说明关系时优先改用普通带标签箭头 `A -->|关系说明| B`，或把说明写在图下方文字中。
- 各历史阶段、王朝、政权或重要时期单独成篇，放在同一目录下。
- 阶段笔记一般使用以下结构：`# 名称`、`## 时间`、可选的 `## 别称`、`## 概括`、`## 说明`、`## 演变关系`。
- `## 说明` 用条目保留具体事实、人物、事件、迁都、灭亡、称臣、战争等细节；不要把这些细节放在目录 `README.md` 里。
- `## 演变关系` 写清前一节点、后一节点、并列关系或分支关系，并链接到同目录相关笔记。
- 同一时期并列的小政权或部落联盟不要过度拆碎；可以合并为一篇综合笔记，并在笔记内部用小标题区分。例如“三韩”合并马韩、辰韩、弁韩，“后三国”合并后百济、后高句丽 / 泰封。
- 从图片、流程图或旧笔记转写内容时，不写“原图”“图片参考”“流程图参考”等来源性话语；直接整理成知识笔记，并确保文字信息不丢失。
- 政体节点笔记应尽量加入统治结构表；君主、国家元首、政府首脑、行政首脑、实际最高领导人按角色分成不同表格，不混在同一张表里。

## 君主与世系表约定

- 君主制或王朝类节点的君主世系表必须严格按在位时间顺序排列，并尽量完整覆盖该政体的全部君主。
- 整理中国政权、王朝、割据政权的君主世系时，世系表字段应尽量包含：顺序、姓名、庙号、谥号、年号（可多个）、在位时间（可多次）、生卒时间、与前任关系、关键事件 / 备注 / 说明。某项无正式称号或资料不明时，写“无”“不详”或简短说明，不要留空。
- 世系表不长，或原笔记内容不多时，直接在原笔记中添加世系表。
- 世系表很长，且原笔记已有大量其他内容时，单独建立“君主世系表 / 王世系表”笔记；主笔记只保留简短摘要并链接到该表。
- 如果原笔记很长且没有专属目录，应先创建同名子目录，把原笔记移入并改名为 `README.md`；再在同层建立世系表笔记，并在 `README.md` 中引用它。
- 世系表中可以用 Markdown 粗体高亮少量关键君主，例如建国者、制度改革者、扩张或统一关键人物、转折期君主、末代君主；不要把普通继承者全部加粗。
- 如果某一时期实际由某个家族长期统治，可以单独建立该家族或该政体的统治者世系笔记，避免在多个主笔记中重复维护同一批人物。

## Codex / AI 操作约定

- 操作这个 vault / repo 时，读取、搜索、创建、移动、重命名、修改笔记都优先使用 Obsidian MCP。
- 只有当 Obsidian MCP 无法完成任务，或任务明显不属于 Obsidian 笔记操作时，才使用其他工具。
- 所有 Git commit message 必须使用中文书写。
- 未得到用户明确允许时，不要修改 Git 暂存区 / index，也不要对已暂存内容执行撤销、取消暂存、覆盖或进一步暂存；禁用命令包括但不限于 `git add`、`git mv`、`git rm`、`git reset`、`git restore --staged`。
- 移动、重命名文件或目录时，默认使用普通文件系统命令 `mv`、`mkdir`、`rmdir`，不要使用 `git mv`；只有用户明确要求“移动并暂存”“stage 这些移动”“提交前整理暂存区”时，才可以使用 `git mv` 或后续 `git add`。
- 执行文件移动前后可以使用 `git status --short` 检查状态，但检查命令不得改变暂存区。
